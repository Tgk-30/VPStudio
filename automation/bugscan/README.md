# VPStudio Bugscan Automation (v2)

Automated bug-scanning workflow using isolated lane-based collector agents,
a validator agent, and a manager agent.

## Architecture

```
automation/bugscan/
├── README.md              # this file
├── findings/              # JSONL findings per lane (append-only)
│   ├── lane-a.jsonl       # Lane A: app/core/models/data-network-services
│   ├── lane-b.jsonl       # Lane B: viewmodels/UI/navigation
│   └── lane-c.jsonl       # Lane C: player/immersive/settings/AI
├── state/                 # JSON state per lane + validator + manager
│   ├── lane-a.json
│   ├── lane-b.json
│   ├── lane-c.json
│   ├── validator.json
│   └── manager.json
├── validation/            # validation events (append-only)
│   └── events.jsonl
├── prompts/               # prompt files for cron jobs
│   ├── collector-a.txt
│   ├── collector-b.txt
│   ├── collector-c.txt
│   ├── validator.txt
│   └── manager.txt
└── scripts/
    ├── import_legacy_report.py   # one-time: bootstrap from markdown
    └── render_report.py          # regenerate marchVPStudioBugFixes.md
```

## Key principles

- **Markdown is output, not database.** The JSONL files are the source of truth.
  The markdown report at `marchVPStudioBugFixes.md` is regenerated by `render_report.py`.
- **Lane isolation.** Each collector reads/writes only its own lane files.
- **Append-only findings.** Collectors never delete findings. The validator marks
  them as duplicate/invalid/superseded via events.jsonl.
- **Manager controls scan pace.** The manager inspects git changes and reactivates
  saturated lanes when their code changes. Collectors just read their scanMode.

## Agents (5 cron jobs)

| Agent         | Prompt file        | Reads                     | Writes                     |
|---------------|--------------------|---------------------------|----------------------------|
| collector-a   | collector-a.txt    | lane-a.json, lane-a.jsonl | lane-a.json, lane-a.jsonl  |
| collector-b   | collector-b.txt    | lane-b.json, lane-b.jsonl | lane-b.json, lane-b.jsonl  |
| collector-c   | collector-c.txt    | lane-c.json, lane-c.jsonl | lane-c.json, lane-c.jsonl  |
| validator     | validator.txt      | all findings, all states  | events.jsonl, validator.json, report |
| manager       | manager.txt        | all lane states, git log  | lane states, manager.json, report |

## Scripts

### import_legacy_report.py

One-time bootstrap: parses `marchVPStudioBugFixes.md` and creates the initial
JSONL findings, validation events, and lane state files.

```bash
python3 automation/bugscan/scripts/import_legacy_report.py
```

### render_report.py

Regenerates `marchVPStudioBugFixes.md` from the machine-readable files.

```bash
python3 automation/bugscan/scripts/render_report.py           # write to file
python3 automation/bugscan/scripts/render_report.py --dry-run  # print to stdout
```

```json
  "scope": "app/core/models/data-network-services",

  "scanMode": "cool",
  "findingCount": 25,
  "ownerModel": "minimax",
  "lastSummary": "...",
  "priorNotes": ["..."]
}
```

## Finding schema (JSONL)

```json
{
  "id": "LANE-A-2026-03-31-A-001",
  "title": "Short descriptive title",
  "confidence": "high",
  "paths": ["VPStudio/path/to/File.swift"],
  "why_it_is_a_bug": "...",
  "trigger_or_repro": "...",
  "impact": "...",
  "evidence": "..."
}
```

```json
{
  "type": "duplicate|invalid|superseded|revalidated",
  "findingId": "LANE-X-...",
  "relatedFindingId": "LANE-Y-...",
  "detail": "...",
  "timestamp": "ISO 8601",
  "source": "validator|legacy_import"
}
```
